* Filosofía general

Este repositorio separa estrictamente /fuentes/ (rama =main=) de
/artefactos generados/ (rama =gh-pages=):

- =main= contiene únicamente ficheros =.org=, figuras =.png=, datos y
  ficheros de configuración (=makefile=, =css/=, =mycfg-GitHubPages.py=,
  =requirements.txt=). Nada generado (pdf, html, ipynb, slides, guiones
  =.inp= de Gretl) se versiona aquí.
- =gh-pages= contiene únicamente lo generado: HTML, PDF, cuadernos
  Jupyter, diapositivas reveal.js y los guiones =.inp= tangled. Se
  reconstruye por completo cada vez que se despliega; no tiene historia
  "significativa" propia, es un espejo del estado actual de =main=.

Esta separación evita mezclar en el historial de =main= cambios de
contenido con cambios puramente mecánicos de regeneración, y mantiene el
repositorio ligero.

* Estrategia B (actual): build local + despliegue manual

** Requisitos en la máquina local

Emacs + scimax, Gretl (=gretlcli=), una distribución LaTeX con soporte
=minted= (=--shell-escape= + =pygments=), y Jupyter con =nbconvert=. Todo
ello ya está operativo en la máquina de trabajo habitual del autor: no
se reinstala nada en cada publicación, a diferencia de lo que ocurriría
con integración continua (ver Estrategia A).

El =makefile= invoca siempre Emacs cargando explícitamente el =init.el=
de scimax (variable =SCIMAX_INIT=, por defecto
=$(HOME)/Software/scimax/init.el=), nunca el =~/.emacs.d/init.el=
genérico del sistema. Esto es necesario porque la configuración de
scimax es la que provee, entre otras cosas, =ox-ipynb= y el resto de
paquetes usados en la exportación.

** Compilar todo con =Makefile=

#+begin_src sh
make all index
#+end_src

La generación de las lecciones depende de que las figuras =.png=
(compiladas a partir del código tikz en =org-lessons/img/*/*.org=)
estén al día. El =makefile= principal invoca automáticamente, como
prerrequisito de =lessons=, el =makefile= auxiliar de
=org-lessons/img/= (que decide por timestamps qué figuras regenerar).
No es necesario ejecutarlo aparte.

=make clean= limpia también los =.tex=/=.aux=/=.log= residuales de la
exportación LaTeX, las carpetas =_minted-*= y las carpetas de salida
que =gretlcli= crea junto a cada práctica ejecutada (mismo nombre que
el =.org=, sin extensión). Tras =make clean= el árbol de =main= debe
quedar reducido a fuentes y ficheros de configuración.

=make all index= genera, dentro de =org-lessons/=, =org-pract/= y
=CuadernosElectronicos/=, todos los PDF, HTML, =.ipynb= y diapositivas, y
tangla + ejecuta los guiones de Gretl de las prácticas. También genera
=index.html= a partir de =index.org=.

*** Por qué las reglas de PDF de =org-pract= ejecutan el buffer antes de exportar

Las reglas =%.pdf= y =%.html= de =org-pract/= dependen de un fichero
/stamp/ =%.executed= (que a su vez depende de =%.tangled=), pero
además cada una de ellas vuelve a invocar =org-babel-execute-buffer=
justo antes de exportar, en el mismo proceso Emacs que hace la
exportación. Esto no es redundante ni un descuido: es imprescindible.

Cada regla de =make= que invoca =$(EMACS_BATCH)= arranca un /proceso
Emacs independiente/. Ninguna variable Elisp fijada en el preámbulo de
un =.org= (por ejemplo =org-latex-minted-langs=, =org-latex-listings= o
=org-latex-pdf-process=, que cada práctica fija en un bloque
=emacs-lisp= de su preámbulo para que =minted= resalte correctamente el
código "gretl" mapeándolo al lexer =octave= de Pygments) sobrevive
entre procesos distintos. Si la exportación a PDF se hiciera en un
proceso batch nuevo que solo hace =find-file= + =org-latex-export-to-pdf=
sin re-ejecutar antes ese preámbulo, las variables nunca se fijarían
en /ese/ proceso y los recuadros de código saldrían vacíos, aunque el
PDF se genere sin error alguno (véase "Incidentes conocidos", más
abajo, para la anécdota completa de cómo se detectó esto).

** Preparar la rama =gh-pages= (una sola vez, ya hecho)

#+begin_src sh :eval no
git checkout --orphan gh-pages
git rm -rf --cached .
git commit --allow-empty -m "Rama gh-pages: sitio publicado (contenidos generados)"
git push origin gh-pages
git checkout main
git worktree add _ghpages gh-pages
#+end_src

El directorio =_ghpages/= es un /worktree/ de git: la misma repositorio,
pero con la rama =gh-pages= pagada en esa carpeta. Permite tener =main= y
=gh-pages= abiertos simultáneamente sin cambiar de rama. Esta carpeta está
en =.gitignore= de =main= (no debe versionarse ahí).

** Desplegar la rama =gh_pages= con el script =./deploy-gh-pages.sh=

#+begin_src sh
./deploy-gh-pages.sh
#+end_src

El script =deploy-gh-pages.sh= reconstruye =_ghpages/= por completo en
cada ejecución (nunca acumula sobre el despliegue anterior) y clasifica
los artefactos generados en subcarpetas: =Lecciones-pdf/=,
=Lecciones-html/=, =Transparencias/= (slides), =Practicas-pdf/=,
=Practicas-html/= (+ =guiones/= con los =.inp=) y
=CuadernosElectronicos/= (solo los de =CuadernosElectronicos/*.org=,
nunca los =.ipynb= intermedios de =org-lessons=). Copia también
=index.html= y crea =.nojekyll=.

Además —y esto es imprescindible para que el material publicado se
vea completo, no solo se genere sin error— el script copia:
- =org-lessons/img/= dentro de =Lecciones-html/= y de =Transparencias/=,
  ya que tanto el HTML de las lecciones como los slides reveal.js
  referencian las figuras con ruta relativa =img/<subcarpeta>/*.png=.
- La carpeta de salida de cada práctica (=org-pract/<nombre>/=, con los
  gráficos y resultados que genera =gretlcli= al ejecutar el =.inp=)
  dentro de =Practicas-html/=, con el mismo nombre que el =.org=
  correspondiente (sin extensión), excluyendo =guiones/= (que ya se
  trata aparte).

*** Salvaguardas del script (importantes: leer antes de tocar el script)

- Determina su propia ubicación con =dirname "${BASH_SOURCE[0]}"=, no
  con =git rev-parse --show-toplevel= (este último es ambiguo cuando se
  invoca desde dentro de un /worktree/ secundario como =_ghpages=). Por
  ello es indiferente ejecutarlo como =./deploy-gh-pages.sh= desde la
  raíz de =main= o como =../deploy-gh-pages.sh= desde dentro de
  =_ghpages=.
- Comprueba, antes de tocar nada, que =main= no está por error en la
  rama =gh-pages=, que =_ghpages/.git= existe (usa =-e=, no =-d=: en un
  /worktree/ =.git= es un fichero, no un directorio) y que =_ghpages=
  está efectivamente en la rama =gh-pages=. Si algo no cuadra, aborta
  con un mensaje explicativo en vez de continuar.
- /Nunca/ usa =git clean -fdx= (con =-x=): solo =-fd=. La opción =-x=
  ignora =.gitignore= y borra también lo ignorado; es la causa de dos
  incidentes de pérdida de ficheros personales (véase más abajo,
  "Incidentes y lecciones aprendidas").

*** Limitaciones conocidas

**** El código "gretl" no sale resaltado con color en las exportaciones HTML

En los PDF, el código de los bloques =#+begin_src gretl= sí sale
coloreado, porque el preámbulo de cada práctica mapea explícitamente
=gretl= al lexer =octave= de Pygments (=org-latex-minted-langs=), y
=pygmentize= sí dispone de ese lexer.

En HTML, en cambio, /ningún/ bloque de código sale coloreado
—incluidos lenguajes con soporte nativo como =python= o =emacs-lisp=,
no solo =gretl=—, aunque sí conservan atributos como negrita o cursiva.
Se investigó la causa: Org exporta código a HTML apoyándose en
=htmlize=, que a su vez consulta =(display-color-p)= para decidir si el
entorno soporta color. En un proceso Emacs =--batch= (como el que usa
este =makefile= para toda la generación) esa función devuelve =nil= (no
hay pantalla), así que =htmlize= omite el color aunque sí aplique otros
atributos de la cara.

Se probó forzar =(advice-add 'display-color-p :override (lambda (&optional _) t))=
antes de exportar, sin éxito: el color sigue sin aparecer. No se ha
seguido investigando más allá porque el beneficio es puramente
cosmético en un formato secundario (el PDF, que es la versión "seria"
para imprimir/estudiar, sí muestra el código resaltado correctamente).
Si en el futuro se quiere retomar, el punto de partida sería revisar
=org-html-fontify-code= en =ox-html.el= para ver qué otra condición,
además de =display-color-p=, pueda estar impidiendo la generación de
=<span style="color:...">=.

**** El lenguaje "gretl" no existe como tal ni en Pygments ni en Emacs

"gretl" no tiene lexer propio en Pygments ni major-mode propio en
Emacs. Para el PDF se resuelve mapeándolo al más parecido disponible,
=octave=, mediante =(add-to-list 'org-latex-minted-langs '(gretl "octave"))=
en el preámbulo de cada práctica. Si algún día aparece un lexer real
para gretl en Pygments, basta con sustituir esa línea.

* Estrategia A (futura, si se desea automatizar con CI)

Si en el futuro se prefiere que la publicación ocurra automáticamente en
cada =push= a =main= (sin depender de ejecutar el script a mano), la vía
es GitHub Actions. Se documenta aquí para no tener que rediseñarlo desde
cero, pero /no está activada/ por defecto.

El fichero =.github/workflows/build.yml.disabled= (más abajo) ya
incorpora las mismas correcciones que =deploy-gh-pages.sh= (copia de
=img/= y de las carpetas de salida por práctica). Antes de activarlo
conviene además:
- Aclarar la discrepancia observada en compilación local entre el
  motor LaTeX pedido (=-xelatex= en =org-latex-pdf-process=) y el que
  =latexmk= termina invocando según el log (=lualatex=); por precaución
  el workflow instala ambos (=texlive-xetex= y =texlive-luatex=).
- Confirmar si la limitación de resaltado HTML documentada arriba
  también se reproduce en el entorno de CI (es esperable que sí, por
  la misma razón: proceso Emacs =--batch= sin pantalla).

** Coste y riesgos a tener en cuenta

- Instalar Emacs + scimax + Gretl + una distribución LaTeX completa en
  cada ejecución del workflow es lento (varios minutos) y fràgil: scimax
  puede intentar compilar paquetes de MELPA la primera vez.
- No hay reconstrucción incremental real entre ejecuciones de CI (el
  =checkout= siempre parte limpio), aunque el =Makefile= use timestamps.
- Mejora natural: empaquetar toda la cadena de herramientas (Emacs,
  scimax, Gretl, TeXLive, Jupyter) en una /imagen Docker/ propia,
  publicada en un registro (Docker Hub o GitHub Container Registry), y
  hacer que el =job= de Actions use esa imagen como =container:=. Esto
  evita reinstalar dependencias en cada ejecución y hace reproducible el
  entorno de compilación.

** Pasos para migrar a Estrategia A cuando se decida

1. Crear un =Dockerfile= con Emacs, scimax preinstalado y cacheado,
   Gretl, TeXLive (subconjunto: =texlive-xetex texlive-luatex
   texlive-latex-extra texlive-fonts-extra latexmk=), =pygments= y
   Jupyter/nbconvert.
2. Publicar la imagen (p. ej. =ghcr.io/mbujosab/peconometria-build:latest=).
3. Activar el fichero =.github/workflows/build.yml= (ya escrito más abajo,
   /listo para usar/ una vez exista la imagen) cambiando el paso de
   instalación de dependencias por =container: ghcr.io/mbujosab/peconometria-build:latest=.
4. Verificar en un =workflow_dispatch= manual antes de dejarlo disparado
   por =push= a =main=.

** =.github/workflows/build.yml= (preparado, no activo por defecto)

#+begin_src yaml :tangle .github/workflows/build.yml.disabled
name: Build y despliegue a gh-pages

on:
  push:
    branches: [main]
    paths:
      - 'org-lessons/**'
      - 'org-pract/**'
      - 'CuadernosElectronicos/**'
      - 'index.org'
  workflow_dispatch:

permissions:
  contents: write

jobs:
  build:
    runs-on: ubuntu-latest
    # Cuando exista la imagen Docker propia, sustituir el paso de
    # "Instalar dependencias" por:
    #   container: ghcr.io/mbujosab/peconometria-build:latest
    steps:
      - uses: actions/checkout@v4

      - name: Instalar dependencias
        run: |
          sudo apt-get update
          sudo apt-get install -y emacs-nox gretl \
            texlive-xetex texlive-luatex texlive-latex-extra \
            texlive-fonts-extra latexmk
          pip install jupyter nbconvert pygments

      - name: Clonar scimax
        run: git clone --depth 1 https://github.com/jkitchin/scimax.git $HOME/Software/scimax

      - name: make all
        run: make all index

      - name: Preparar sitio
        run: |
          mkdir -p _site/{Transparencias,Lecciones-html,Lecciones-pdf,Practicas-html/guiones,Practicas-pdf,CuadernosElectronicos}

          find org-lessons -maxdepth 1 -name '*.slides.html'                  -exec cp {} _site/Transparencias/ \;
          find org-lessons -maxdepth 1 -name '*.html' ! -name '*.slides.html' -exec cp {} _site/Lecciones-html/ \;
          find org-lessons -maxdepth 1 -name '*.pdf'                         -exec cp {} _site/Lecciones-pdf/ \;
          find org-lessons -maxdepth 1 -name '*.ipynb'                       -exec cp {} _site/CuadernosElectronicos/ \;

          # Figuras: imprescindibles para ver algo en Lecciones-html y Transparencias
          cp -r org-lessons/img _site/Lecciones-html/
          cp -r org-lessons/img _site/Transparencias/

          find org-pract -maxdepth 1 -name '*.html' -exec cp {} _site/Practicas-html/ \;
          find org-pract -maxdepth 1 -name '*.pdf'  -exec cp {} _site/Practicas-pdf/ \;
          find org-pract/guiones -maxdepth 1 -name '*.inp' -exec cp {} _site/Practicas-html/guiones/ \;

          # Salida de gretlcli por práctica (gráficos, estimaciones, etc.)
          for d in org-pract/*/; do
            nombre=$(basename "$d")
            [ "$nombre" = "guiones" ] && continue
            cp -r "$d" _site/Practicas-html/
          done

          find CuadernosElectronicos -maxdepth 1 -name '*.ipynb' -exec cp {} _site/CuadernosElectronicos/ \;

          cp requirements.txt _site/
          cp index.html _site/
          touch _site/.nojekyll

      - name: Desplegar a gh-pages
        uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./_site
          publish_branch: gh-pages
#+end_src

* Incidentes conocidos y lecciones aprendidas

- =.gitignore= no protege de =git clean -fdx=: la opción =-x= ignora
  deliberadamente las reglas de =.gitignore= y borra también lo
  ignorado. Cualquier script de limpieza en este repositorio debe usar
  =-fd= a secas, nunca =-x=.
- Los ficheros verdaderamente irremplazables y personales (p. ej. el
  plan de escritura del curso) no deben vivir /dentro/ del árbol de
  este repositorio ni siquiera como "ignorados": deben residir en una
  carpeta hermana, fuera del repo, idealmente con su propio control de
  versiones. =.gitignore= es una convención, no una barrera física.
- Si el árbol de trabajo está accesible a través de un enlace
  simbólico (p. ej. dos clientes de sincronización en la nube apuntando
  al mismo directorio con nombres distintos), los /worktrees/ de Git
  pueden quedar con rutas absolutas internas inconsistentes. Si
  =git worktree list= muestra el worktree pero cualquier operación
  sobre él falla, probar primero =git worktree repair= antes de asumir
  pérdida de datos.
- Cualquier script que borre contenido debe /verificar activamente/
  (rama actual, existencia y naturaleza del destino) antes de actuar, y
  abortar con un mensaje claro si algo no es exactamente lo esperado,
  en vez de confiar en que "se invocará siempre bien".
- Cada regla de =make= que invoca =$(EMACS_BATCH)= arranca un proceso
  Emacs independiente; ninguna variable Elisp fijada en un preámbulo
  =.org= sobrevive entre reglas encadenadas por ficheros stamp. Cualquier
  exportación que dependa de configuración establecida por bloques del
  propio documento (como el mapeo =gretl→octave= para minted) debe
  ejecutar =org-babel-execute-buffer= /en el mismo proceso/ que hace la
  exportación, no confiar en pasos previos ya cerrados. Este fue el bug
  que causaba que los PDF de =org-pract= se generasen sin error pero con
  los recuadros de código de minted completamente vacíos.
- El resaltado de sintaxis en HTML (vía =htmlize=) depende de
  =(display-color-p)=, que en un proceso Emacs =--batch= (sin pantalla)
  devuelve =nil=. Por eso ningún bloque de código sale coloreado en las
  exportaciones HTML de este repositorio, con independencia del
  lenguaje. Se documenta como limitación conocida (ver "Limitaciones
  conocidas") en lugar de como bug pendiente, dado el coste/beneficio
  de seguir investigando frente al hecho de que el PDF ya muestra el
  código correctamente resaltado.
